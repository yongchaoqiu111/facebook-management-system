## 2. 核心业务规则：六合红包（⭐ 必读）

在开始开发前，请务必理解以下业务逻辑，否则会导致接口报错。

### 2.1 群组限制
**六合红包只能在「六合天下」群发送！**
- **群组 ID**: `69d3f89668f596338b0c1930`
- **前端逻辑**: 只有当用户进入该群组时，才显示“发六合红包”的入口。在其他群或私聊中隐藏该功能。

### 2.2 时间段控制
每天分为三个时段，前端需根据时间显示不同的 UI 状态：
- 🟢 **投注时段 (00:00 - 20:32)**：允许发红包、允许投注。
- 🟡 **封闭等待 (20:33 - 21:32)**：禁止任何操作，显示倒计时等待开奖。
- 🔵 **结算时段 (21:33 - 23:59)**：系统自动开奖并结算，禁止操作。

### 2.3 赔率与风控
- **号码范围**: 1-49。
- **赔率**: 49倍（平台抽 1 倍，玩家实得 48 倍）。
- **单号上限**: 动态计算 = 奖池 ÷ 48。例如奖池 490 USDT，则每个号码最多投 10.21 USDT。

## 3. 关键开发规范（避坑指南）

### 3.1 API 代理配置（重要）
为了简化代码并避免跨域问题，请在 `vite.config.js` 中配置代理：
```javascript
export default defineConfig({
  server: {
    proxy: {
      '/api': {
        target: 'http://localhost:5000', // 指向后端
        changeOrigin: true
      }
    }
  }
})
```

### 3.2 ID 使用规范（❌ 严禁踩坑）
- **必须使用 MongoDB ObjectId**: 所有的业务 ID（如红包 ID、群 ID）必须从后端 API 返回或 Socket 推送中获取。
- **格式示例**: `69d4063546d61dd912853593`（24位十六进制字符串）。
- **严禁使用临时 ID**: 绝对不要在前端生成类似 `redpacket_${Date.now()}` 的 ID 传给后端，这会导致 `500 Internal Server Error`。

### 3.3 实时通信 (Socket.io)
- **房间机制**: 进入群组时，确保前端已加入对应的 Socket 房间（通常是 `group:${groupId}`）。
- **监听事件**: 
  - `newLiuheRedPacket`: 收到新红包时刷新列表。
  - `liuheBetUpdate`: 投注额度变化时更新 UI。
  - `liuheResult`: 开奖结果推送。

## 10. 关键 API 接口

### 10.1 用户相关
- `POST /api/auth/login` - 用户登录
- `POST /api/auth/register` - 用户注册

### 10.2 聊天相关
- `GET /api/chat/messages/:userId` - 获取聊天记录

- `POST /api/redpackets/create` - 创建普通红包
- `POST /api/liuhe/create` - 创建六合红包
- `POST /api/liuhe/:id/bet` - 六合红包投注
- `POST /api/redpackets/:id/open` - 打开红包

### 10.4 钱包相关
- `GET /api/wallet/balance` - 获取余额
- `POST /api/wallet/deposit` - 充值
- `POST /api/wallet/withdraw` - 提现

### 10.5 六合彩相关
- `GET /api/lottery/latest` - 获取最新开奖结果

## 15. 常见报错处理 (FAQ)

1. **404 Not Found**: 
   - 检查 `vite.config.js` 是否配置了 `/api` 代理。
   - 检查请求路径是否拼写正确。
2. **500 Internal Server Error**: 
   - 90% 的情况是因为传入了错误的红包 ID（如 `redpacket_123`）。请打印出你传入的 ID 进行检查。
3. **CastError**: 
   - 同上，后端无法将非 ObjectId 格式的字符串转换为数据库查询条件。
